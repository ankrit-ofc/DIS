# DISTRO Deployment Guide

Last updated: 2026-04-27

## Architecture

- **Frontend:** Next.js 14 (`distro-web/`) → Vercel
- **API:** Express + Prisma (`distro-api/`) → Railway
- **Database:** MySQL → Railway managed plugin
- **File uploads:** local disk (`UPLOAD_DIR=./uploads`) — Railway disks are ephemeral, so files DO NOT survive redeploys. Migrate to Cloudflare R2 before relying on uploaded media.
- **Domain:** `distronepal.com` → Vercel; `api.distronepal.com` → Railway

## Pre-deployment checklist

- [ ] Code pushed to GitHub `main` branch
- [ ] Prisma migrations committed (`distro-api/prisma/migrations/` + `migration_lock.toml`)
- [ ] `.env.example` reflects current env vars in both apps
- [ ] `package.json` has `build`, `start`, `db:deploy` scripts in `distro-api`; `build`, `start` in `distro-web`
- [ ] No `.env` file committed (verify with `git ls-files | grep -E "\.env($|\.)" | grep -v example`)
- [ ] No real secrets in committed files
- [ ] Local build succeeds: `npm run build` in both apps

## Step 1 — Railway: Database + API

1. Sign up at railway.app (use GitHub auth).
2. **New Project → "Provision MySQL"**. Wait for the plugin to spin up.
3. Click the MySQL service → **Variables** tab → copy `MYSQL_URL` (this becomes `DATABASE_URL` for the API).
4. **New Service** in the same project → **Deploy from GitHub repo** → select your fork.
5. Service settings:
   - **Root directory:** `distro-api`
   - **Build command:** `npm install && npm run build`
     (Prisma client is auto-generated by the `postinstall` hook.)
   - **Start command:** `npm run db:deploy && npm start`
     (`db:deploy` runs `prisma migrate deploy` — applies pending migrations without prompts. Safe to run on every boot.)
6. **Variables** tab → add every entry from `distro-api/.env.example` with REAL values:
   - `DATABASE_URL` — paste the value from step 3 (or use Railway's reference variable: `${{ MySQL.MYSQL_URL }}`)
   - `JWT_SECRET` — generate fresh: `openssl rand -hex 32`
   - `ADMIN_SEED_PASSWORD` — strong password (used only on first seed)
   - `NODE_ENV=production`
   - `CORS_ORIGIN=https://distronepal.com` (update with Vercel URL after step 2)
   - `SMS_ENABLED=true`, `SPARROW_SMS_TOKEN=<rotated>`, `SPARROW_SMS_FROM=<sender-id>`
   - `EMAIL_ENABLED=true`, `RESEND_API_KEY=<rotated>`, `RESEND_FROM=<verified-domain>`, `ADMIN_EMAIL`
   - `GOOGLE_CLIENT_ID`
   - `COMPANY_NAME`, `COMPANY_ADDRESS`, `COMPANY_PAN`, `VAT_RATE`
7. Deploy. Watch logs for: `Prisma migrate deploy` success → `API listening on :3001`.
8. Copy the public Railway URL (e.g., `distro-api-production.up.railway.app`).
9. Smoke test: `curl https://<railway-url>/api/health` (or whichever health path exists).
10. **First-time seed.** From your local terminal, pointed at the Railway DB:
    ```bash
    cd distro-api
    DATABASE_URL="<railway-mysql-url>" SEED_RESET=1 ADMIN_SEED_PASSWORD="<strong-pwd>" NODE_ENV=production npm run db:seed
    ```

## Step 2 — Vercel: Frontend

1. Sign up at vercel.com (use GitHub auth).
2. **New Project** → import the repo.
3. Project settings:
   - **Root directory:** `distro-web`
   - **Framework:** Next.js (auto-detected)
   - **Build/Install/Output:** keep defaults.
4. **Environment Variables** — add from `distro-web/.env.example`:
   - `NEXT_PUBLIC_API_URL=https://<railway-url>/api` (note the `/api` suffix)
   - `NEXT_PUBLIC_GOOGLE_CLIENT_ID=<value>`
   - `NEXT_PUBLIC_GOOGLE_MAPS_API_KEY=<value>` (restrict by HTTP referrer in Google Cloud Console)
5. Deploy. Watch build logs.
6. Copy the Vercel public URL (e.g., `distro-web.vercel.app`).
7. Smoke test: open the URL → catalogue loads, login works, place test order.

## Step 3 — Update CORS to allow the Vercel URL

In Railway → distro-api service → **Variables** → `CORS_ORIGIN`:
```
https://distronepal.com,https://<vercel-url>.vercel.app
```
The API parses comma-separated origins (`distro-api/src/app.ts:35`). Trigger a redeploy.

## Step 4 — DNS

1. In your registrar (Nest Nepal panel for `distronepal.com`):
   - Point apex `@` to Vercel (Vercel → Domains tab will show the IP / instructions).
   - `CNAME www` → `cname.vercel-dns.com`.
2. Optionally add `CNAME api` → your Railway URL (or use Railway's custom-domain flow).
3. Wait 1–4 hours for DNS to propagate. Vercel + Railway auto-issue SSL certs once DNS resolves.
4. After `api.distronepal.com` resolves, update `NEXT_PUBLIC_API_URL` on Vercel to use it (and `CORS_ORIGIN` on Railway accordingly), then redeploy both.

## Step 5 — Post-deployment checklist

- [ ] `https://distronepal.com` loads homepage
- [ ] Buyer signup works (OTP via SMS arrives)
- [ ] Login works for all 3 roles (buyer, admin, driver) at `/login`
- [ ] Place a real test order — admin sees it, can confirm, PDF invoice generates
- [ ] Order email arrives in customer inbox (Resend domain must be verified)
- [ ] Driver dashboard at `/driver` works, Google Maps navigation opens
- [ ] PWA installs on iPhone (Add to Home Screen → DISTRO logo + working layout)
- [ ] Rotate every credential ever pasted in chat/dev:
  - [ ] `JWT_SECRET` (regenerated)
  - [ ] `SPARROW_SMS_TOKEN`
  - [ ] `RESEND_API_KEY`
  - [ ] `GOOGLE_MAPS_API_KEY` (and add referrer restriction)
  - [ ] Admin password (changed via UI, not via env)
  - [ ] Database password (Railway-managed, regenerate via plugin if exposed)

## Known issues to address post-launch

- **File uploads on Railway disk** — ephemeral. Migrate to Cloudflare R2 before users upload anything you can't lose.
- **Resend `RESEND_FROM`** — currently uses a Gmail address in dev. Must use a verified domain (`distronepal.com`) in production, otherwise mail will be rejected/junked.
- **`MIN_ORDER_VALUE`** — duplicated between code and DB Setting; consolidate post-launch.
- **Mobile (Expo) deployment** — separate process; not covered here.

## Rolling back

- **Vercel:** Dashboard → Deployments → previous one → "Promote to production".
- **Railway:** Service → Deployments → click an older deployment → "Redeploy".
- **DNS:** revert apex to the previous host (Nest Nepal IP) only as a last resort — DNS takes hours to propagate back.
